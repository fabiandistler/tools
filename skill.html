Was ist ein Agent Skill?

Ein Agent Skill ist ein wiederverwendbares Wissenspaket, das einem LLM-Agenten beibringt, eine bestimmte Klasse von Aufgaben zuverlässig auszuführen — ohne es jedes Mal neu erklären zu müssen.

📥
Trigger User-Anfrage matcht die Skill-Description
→
📖
Load SKILL.md wird in den Kontext geladen
→
⚙️
Execute Claude folgt den Instruktionen
→
📦
Output Deterministisches, hochwertiges Ergebnis

📁 Struktur

Jeder Skill ist ein Ordner mit einer SKILL.md plus optionalen Ressourcen (Scripts, References, Assets).

YAML Frontmatter Markdown Body

🎯 Triggering

Claude sieht nur name + description. Die Description ist das primäre Triggering-Mechanismus — sie muss präzise sein.

Spezifisch Aktiv-formuliert

📐 Progressive Disclosure

3 Ladeschichten: Metadata → SKILL.md Body → Bundled Resources. Nur was gebraucht wird, kommt in den Kontext.

~100 Wörter <500 Zeilen
💡 Kernprinzip: Ein Skill löst ein Problem, das Claude alleine nicht zuverlässig lösen kann. Einfache 1-Schritt-Aufgaben brauchen keinen Skill — nur komplexe, mehrstufige oder spezialisierte Workflows profitieren davon.

Dateisystem-Struktur

my-skill/
├── SKILL.md          # Pflicht: YAML frontmatter + Instruktionen
├── scripts/
│   └── process.py    # Ausführbare Scripts für repetitive Aufgaben
├── references/
│   ├── aws.md        # Domain-spezifische Doku (nur bei Bedarf laden)
│   └── gcp.md
└── assets/
    └── template.docx # Dateien für den Output
💡 Principle of Least Surprise: Der Skill soll tun, was seine Description verspricht — nicht mehr, nicht weniger. Keine unerwarteten Seiteneffekte.

Komplexitätsstufen

🟢 Simple Skill

Klar umrissene Aufgabe, ein Output-Format, keine externen Abhängigkeiten, <100 Zeilen SKILL.md.

1 Input-Typ 1 Output-Typ Kein Script 
---
name: pdf-summary
description: |
  Fasst PDF-Dokumente präzise zusammen.
  Nutzen wenn: User sagt "fasse zusammen",
  "tldr", oder eine PDF-Datei hochlädt und
  nach dem Inhalt fragt.
---

# PDF Summary Skill

## Vorgehen
1. Lese das komplette PDF
2. Identifiziere: Thema, Kernaussagen, Fazit
3. Schreibe eine strukturierte Zusammenfassung

## Output-Format
- **Thema:** 1 Satz
- **Kernpunkte:** 3–5 Bullet Points  
- **Fazit:** 2–3 Sätze

## Constraints
- Keine Zitate über 15 Wörter
- Neutral, keine Wertung
- Immer Deutsch, auch wenn PDF englisch
name Kurz, lowercase, mit Bindestrich. Eindeutiger Identifier.
description Enthält explizite Trigger-Phrasen ("fasse zusammen", "tldr"). Claude matcht darauf.
Vorgehen Nummerierte Schritte = deterministischer Ablauf.
Output-Format Explizites Schema verhindert kreative Ausreißer.
Constraints Grenzen helfen Claude, Scope zu halten.

🟡 Medium Skill

Mehrere Inputs/Outputs, ein optionales Script, 100–300 Zeilen, mit Entscheidungslogik und Fehlerbehandlung.

Entscheidungslogik 1 Script Fehlerbehandlung 
---
name: docx-report-creator
description: |
  Erstellt professionelle Word-Reports aus
  Daten oder Text. Nutzen bei: "erstelle
  einen Bericht", "Word-Dokument", ".docx",
  "Report für den Kunden". Auch wenn der
  User nach Tabellen oder Formatierung fragt.
---

# DOCX Report Creator

## Lies zuerst
Lese `/mnt/skills/public/docx/SKILL.md`
bevor du Code schreibst.

## Entscheidungslogik

Prüfe den Input:
- Strukturierte Daten (CSV/JSON) → Tabellen
- Freitext → Prose mit Headings
- Gemischt → Hybrid-Layout

## Workflow
1. Input analysieren (Typ, Umfang)
2. Gliederung vorschlagen → User bestätigt
3. Script ausführen: `scripts/generate.py`
4. File nach `/mnt/user-data/outputs/`
5. `present_files` aufrufen

## Fehlerbehandlung
- Fehlende Daten → nachfragen, nicht raten
- Script-Fehler → Fallback auf reinen Text
- Unklarer Scope → immer zuerst klären
description "Auch wenn..." — aktive Formulierung erhöht Trigger-Rate. Mehrere Phrasen-Varianten.
Lies zuerst Verweist auf weiteres Skill-File. Spart Platz in der SKILL.md.
Entscheidungslogik Wenn-Dann-Regeln machen den Skill robuster als eine Einheitslösung.
Workflow Schritt 2 enthält User-Checkpoint — verhindert falsche Annahmen.
Fehlerbehandlung Explizite Fallbacks = kein stilles Scheitern.

🔴 Complex Skill

Multi-Domain, mehrere Scripts, eigene Reference-Files, sub-skill Verweise, iterativer Loop, 300–500 Zeilen SKILL.md.

Multi-Domain Sub-Agents Iterativer Loop Reference-Files 
---
name: salary-analytics-suite
description: |
  Vollständige Gehaltsanalyse-Pipeline:
  Anomalie-Erkennung, Benchmarking, Report.
  Nutzen bei: Gehalts-/Kompensationsanalyse,
  HR-Analytics, Benchmarking, Outlier-Detection
  in Vergütungsdaten, Equity-Analysen.
---

# Salary Analytics Suite

## Überblick
Dieser Skill orchestriert 3 Sub-Workflows.
Entscheide anhand des User-Inputs, welcher:

| Input | Workflow |
|-------|---------|
| Rohdaten + "Ausreißer" | anomaly.md |
| Marktdaten + "Benchmark" | benchmark.md |
| Ergebnisse vorhanden | report.md |

## Reference Files (bei Bedarf laden)
- `references/anomaly.md` — Isolation Forest, Z-Score
- `references/benchmark.md` — Percentile-Methoden
- `references/report.md` — Output-Templates

## Scripts
- `scripts/detect_outliers.R` — Anomalie-Detection
- `scripts/benchmark_calc.R` — Benchmark-Kennzahlen
- `scripts/render_report.R` — PDF/DOCX Output

## Iterativer Loop
1. Daten laden + validieren
2. User: Scope bestätigen (welche Positionen?)
3. Analyse ausführen
4. Ergebnisse präsentieren → Feedback
5. Auf Wunsch: iterieren oder exportieren

## Qualitäts-Gates
- n < 5 pro Gruppe → kein Output, Warnung
- Fehlende Felder >20% → User warnen
- Immer Konfidenz-Intervalle angeben
Routing-Tabelle Klar definierter Entscheidungsbaum — Claude weiß immer, welchen Pfad es nimmt.
Reference Files Werden NUR geladen wenn nötig. Spart Kontext-Budget.
Scripts Deterministischer Code für repetitive Berechnungen — nicht Claude überlassen.
Iterativer Loop Explizite User-Checkpoints verhindern, dass Claude zu weit läuft.
Qualitäts-Gates Verhindert falsche Ergebnisse bei schlechten Daten. Statistik-Ethik.
⚠️ Ab ~500 Zeilen: Füge eine weitere Hierarchie-Ebene hinzu und verweise klar auf Sub-Files. Die SKILL.md wird sonst zur Blackbox.

Best-Practice Patterns

Bewährte Muster für verschiedene Skill-Situationen.

🎯 Gute Description schreiben

Die Description ist der Trigger-Mechanismus. Sie muss spezifisch und aktiv formuliert sein.

# Schlecht — zu vage:
description: >
  Hilft bei Dokumenten.

# Gut — spezifisch + Trigger-Phrasen:
description: |
  Erstellt Word-Reports aus Daten.
  Nutzen wenn: "Bericht erstellen",
  "Word-Dokument", ".docx", "Report"
  oder Tabellen-Formatierung gefragt.

📋 User-Checkpoints einbauen

Bei mehrstufigen Workflows immer Bestätigungs-Punkte einbauen, bevor Claude irreversible Schritte ausführt.

## Workflow
1. Analysiere Input
2. Schlage Gliederung vor → warte auf OK
3. Erstelle Dokument
4. Zeige Vorschau → warte auf Feedback
5. Exportiere final
💡 Schritt 2 und 4 sind Checkpoints — nie ohne User-OK weiter.

🗂 Progressive Disclosure

Nur das in die SKILL.md schreiben, was Claude immer braucht. Alles andere auslagern.

# In SKILL.md:
## API-Referenz
Für Details zur REST-API:
`references/api-spec.md`

## Lade wenn:
- User fragt nach spez. Endpoint
- Fehler beim API-Call auftreten

⚠️ Fehlerbehandlung

Jeder Skill braucht explizite Fallbacks. Was passiert wenn Input fehlt, Script crasht, Daten ungültig sind?

## Fehlerbehandlung
- Fehlende Pflichtfelder → fragen, nicht raten
- Script-Fehler → Fallback beschreiben
- Unklarer Scope → klären vor Ausführung
- n < Mindestgröße → warnen + stoppen

🔀 Routing-Tabellen

Für Multi-Domain-Skills: immer eine klare Entscheidungstabelle an den Anfang stellen.

## Workflow-Auswahl

| Wenn User sagt | Dann nutze |
|----------------|------------|
| "Ausreißer"    | detect.md  |
| "Benchmark"    | bench.md   |
| "Report"       | report.md  |

🧪 Testbarkeit einplanen

Formuliere den Skill so, dass er mit konkreten Test-Prompts überprüfbar ist. Vage Skills sind schwer zu evaluieren.

## Erwarteter Output
Input: salary_data.csv mit 50 Zeilen
Output:
- outliers.csv (Ausreißer-Liste)
- summary.md (Textbericht)
- plot.png (Visualisierung)

# → klar testbar ✓

Skill Quality Checklist

Klick auf die Punkte, um sie abzuhaken. Alle sollten ✅ sein bevor du deployst.

📝 Spezifikation

Name ist kurz, lowercase, mit Bindestrich (z.B. pdf-summary)
Description enthält konkrete Trigger-Phrasen, die User tatsächlich verwenden
Description ist aktiv formuliert ("Nutzen wenn...", nicht nur "Hilft bei...")
Scope ist klar abgegrenzt — was der Skill NICHT tut, ist implizit oder explizit klar

⚙️ Struktur & Inhalt

SKILL.md Body unter 500 Zeilen (oder: klar auf Sub-Files verweisend)
Workflow ist nummeriert und linear (kein Sprungwirrwarr)
Output-Format ist explizit spezifiziert
Bundled Resources (Scripts, Refs) sind nur bei Bedarf geladen, nicht standardmäßig

🛡 Robustheit

Fehlerbehandlung ist explizit — kein stilles Scheitern
User-Checkpoints vor irreversiblen Schritten eingebaut
Edge Cases sind adressiert (fehlende Daten, kleines n, unklarer Scope)

🧪 Testbarkeit

Mindestens 2 konkrete Test-Prompts existieren oder sind geplant
Erwarteter Output ist klar definiert (sodass "richtig/falsch" prüfbar ist)
Skill wurde mindestens einmal manuell mit einem Test-Prompt verifiziert
